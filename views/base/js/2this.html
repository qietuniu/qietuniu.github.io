# This

# 执行上下文及执行上下文栈

# 执行上下文

执行上下文可以理解为当前代码运行的环境，分为全局上下文和函数上下文。代码运行时进去全局上下文，被调用时进入相应的函数上下文。

# 执行上下文栈

JavaScript 引擎创建了执行栈（LIFO）来管理执行上下文，可以把执行栈认为成一个储存函数调用的栈结构，遵循先进后出的原则。

# 流程

在 JavaScript 代码开始执行时，首先进入全局环境，此时全局上下文被创建并入栈，之后当调用函数时则进入相应的函数环境，此时相应函数上下文被创建并入栈，当处于栈顶的执行上下文代码执行完毕后，则会将其出栈。

function fn2() {
    console.log('fn2')
}

function fn1() {
    console.log('fn1')
    fn2();
}
fn1();

以下流程详细说明上面代码三进三出：

  1. 在全局代码执行前, JS就会创建一个栈来储存管理所有的执行上下文对象
  2. 在全局执行上下文(window)确定后, 将其添加到栈中(压栈push)
  3. fn1被调用，fn1的函数执行上下文创建并将其添加到栈中(压栈push)
  4. fn1中调用fn2，fn2的函数执行上下文创建并将其添加到栈中(压栈push)
  5. fn2执行完毕，将栈顶的对象移除，即fn2函数上下文被移出(出栈pop)
  6. fn1执行完毕，fn1函数上下文被移出(出栈pop)
  7. 当所有的代码执行完后, 栈中只剩window，全局上下文被移除(出栈pop)

JavaScript执行在单线程上, 所以的代码都是排队执行. 栈底永远都是全局上下文，而栈顶就是当前正在执行的上下文

# 构造器调用

构造函数默认返回return（正常情况下不写），当构造器不显式返回任何数据或者返回一个非对象类型当数据时，构造器的this指向new运算符调用函数时返回的对象。 当构造器显式地返回一个Object类型的对象时，运算结果就会返回这个Object类型的对象。

var myClass1 = function() {
    this.name = 'qietuniu'
    return 'shizi'
}
var myClass2 = function() {
    this.name = 'qietuniu'
    return {
        name: 'shizi'
    }
}
var class1 = new myClass1()
var class2 = new myClass2()
console.log(class1.name) // qietuniu
console.log(class2.name) // shizi

# apply与call

# apply

语法： apply(thisObj, [args]) apply()接收两个参数，一个是函数运行的作用域（this），另一个是参数数组。

如果args不是一个有效数组或不是arguments对象，那么将导致一个TypeError，如果没有提供args和thisObj任何一个参数，那么Global对象将用作thisObj

# 手写apply

Function.prototype.myApply = function(context) {
    let ctx = context
    if (typeof context == null) {
        ctx = window
    }
    cxt.fn = this
    const res = arguments[1] ? cxt.fn(...arguments[1]) : cxt.fn()
    delete cxt.fn;
    return res;
}

# call

语法： call(thisObj, arg1, arg2, arg3, arg4, ... ) call()接受一个参数列表

call方法可以用来代替另一个对象调用一个方法，call方法可以将一个函数的对象上下文从初始的上下文改变为thisObj指定的新对象，如果没有提供thisObj参数，那么Global对象被用于thisObj。

# 手写call

Function.prototype.myCall = function(context) {
    let ctx = context
    if (typeof context == null) {
        ctx = window
    }
    const args = Array.from(arguments).slice(1)
    cxt.fn = this
    const res = cxt.fn(...args)
    delete cxt.fn;
    return res;
}

# call、apply 的区别？谁性能更好？

# 区别

  1. 参数书写方式不同

调用函数时，js解释权不会计较形参和实参数量，类型以及顺序的区别。从这个方法来说，apply比call使用率高

  1. call是apply的语法糖

call可以让我们明确知道函数接受多少参数

# 性能

https://jsperf.com/call-apply-segu该网站可测做性能分析。根据操作系统和浏览器内核版本等不同，最后的结果也不一样，但总体来说可以得到的结论是：call方法永远比apply方法执行速度要快。 **舍弃掉apply方法或者用call来模拟apply方法

switch (args.length) {
    case 0:
        g = gen.call(ctx);
        break;
    case 1:
        g = gen.call(ctx, args[0]);
        break;
    case 2:
        g = gen.call(ctx, args[0], args[1]);
        break;
    default:
        g = gen.apply(ctx, args);
}

# bind

语法： fun. bind(thisArg[, arg1[, arg2[, ... ]]]) bind 和 call/apply 一样，都是用来改变上下文 this 指向的，不同的是call/apply 是直接使用在函数上，而 bind 返回一个改变了上下文 this 后的函数不会立即执行。

# 手写

  1. bind方法返回的是一个绑定this后的函数，并且该函数并没有执行，需要手动去调用。
  2. bind方法可以绑定this，传递参数。注意，这个参数可以分多次传递。
  3. 如果bind绑定后的函数被new了，那么此时this指向就发生改变。此时的this就是当前函数的实例。
  4. 构造函数上的属性和方法，每个实例上都有。
Function.prototype.myBind = function(context) {
    let _this = this
    let args1 = Array.from(arguments).slice(1)
    let bindFn = function() {
        let args2 = Array.from(arguments)
        return _this.apply(this instanceof bindFn ? this : context, args1.concat(args2))
    }
    let Fn = function() {}
    Fn.prototype = this.prototype
    bindFn.prototype = new Fn()
    return bindFn
}

# 思考题

function test() {
    console.log(this);
}
test(); // this === window,严格下 undefined

// call
test.call(null); // this === window,严格下 null
test.call(undefined); // this === window,严格下 undefined

// apply
test.apply(null); // this === window,严格下 null
test.apply(undefined); // this === window,严格下 undefined

// bind
test.bind(null)(); // this === window,严格下 null
test.bind(undefined)(); // this === window,严格下 undefined
